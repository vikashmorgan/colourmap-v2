# Architecture

## Defaults

- Server Components by default. Client Components only for interactivity.
- No `src/` directory. Top-level `app/` and `lib/` directly.

## Choosing a Tier

Default to **Tool / Experiment**. Upgrade to **Product** only when you need auth or persistent user data.

### Tool / Experiment

No auth, no DB or local-only. Flat structure. API routes can contain business logic directly.

```
app/
├── api/                        # API routes (logic lives here)
├── page.tsx
└── layout.tsx
lib/                            # Shared utilities and helpers
components/                     # Reusable UI components
```

### Product

Auth, persistent DB, user-facing. API routes are thin orchestrators — business logic in `lib/services/`, data access in `lib/db/queries/`. Never put business logic in route handlers or components.

Route groups `(auth)` and `(app)` to separate public and authenticated surfaces.

```
app/
├── api/                        # API routes (thin orchestrators)
├── (auth)/                     # Public: login, callback
└── (app)/                      # Authenticated surfaces
lib/
├── services/                   # Business logic and orchestration
├── db/
│   ├── client.ts               # Drizzle client singleton
│   ├── schema.ts               # All table definitions
│   └── queries/                # All DB access
└── supabase/
    ├── server.ts               # Server-side client (@supabase/ssr)
    └── client.ts               # Browser-side client
components/                     # Reusable UI components
drizzle/migrations/             # Generated by drizzle-kit
middleware.ts                   # Session refresh only — no business logic
```
